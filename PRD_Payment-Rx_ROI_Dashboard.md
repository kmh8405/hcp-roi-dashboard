## 1. 배경 및 문제 정의

영업팀은 현재 의사(HCP)에 대한 지급(Payment) 활동과 실제 처방(Rx) 실적을 별도의 데이터로 각각 확인하고 있어, 지급 투자 대비 처방 성과(ROI)를 파악하기 어렵다. 두 데이터를 결합하면 지급이 처방에 미치는 영향을 정량적으로 파악하고, 영업 투자의 우선순위를 데이터 기반으로 재조정할 수 있다.

### 보유 데이터 현황

**Part D Prescriber (`part_d_prescriber.csv`)**: 캘리포니아(CA) 지역 처방 데이터, 4,036명의 처방의(NPI), 5개 브랜드(Entresto, Cosentyx Pen/Syringe 계열, Lucentis) 대상. Entresto가 전체 청구건(177,461건)과 비용($178.6M) 대부분을 차지.

**Open Payments (`open_payments.csv`)**: 2022년 프로그램 연도, Novartis 계열사가 CA 중심(전체 지급의 약 99.8%) 7,293명의 의사에게 지급한 내역. 24개 제품, 총 지급액 $1,384,132. 식사/음료, 컨설팅, 강연료, 여행/숙박 등 10개 지급 유형(payment_nature) 포함.

**두 데이터 연결점**: `Prscrbr_NPI` = `covered_recipient_npi` 기준 조인 시 1,677명의 의사가 양쪽 데이터에 모두 존재 (처방 데이터 내 NPI의 약 41.6%).

**제약 사항**:
- Part D는 5개 제품만 포함되어 있어, Open Payments의 24개 제품 중 Entresto·Cosentyx·Lucentis 외 제품(Leqvio, Kisqali 등)은 지급 데이터만 있고 처방 매칭이 불가능함 (PRD 3.4 참고).
- 두 데이터셋의 기준 시점(연도)이 명시적으로 일치하는지 확인 필요 — Part D 파일에 연도 컬럼이 없어 별도 확인 필요.
- 데이터는 CA 지역에 99% 이상 집중되어 있어, 초기 버전은 CA 단일 지역 기준으로 설계.

## 2. 목표

1. **1순위 (MVP 핵심)**: 제품별 지급액 대비 처방액 ROI를 시각화하여, 영업 투자 효율이 높은/낮은 제품과 HCP 세그먼트를 식별.
2. Paid vs Unpaid 의사 그룹 간 처방 패턴 비교를 통해 지급 활동의 처방 기여도를 검증.
3. 고처방/고지급 의사 리스트(Top HCP)를 제공하여 영업 리소스 배분 의사결정을 지원.
4. 지급 유형별·전문과목별 분석으로 어떤 유형의 인터랙션이 처방과 더 강하게 연관되는지 파악.

### 성공 기준 (Success Metrics)
- 1개월 내 MVP 출시 (핵심 KPI 위주로 빠르게 1차 버전 배포).
- 영업 관리자가 대시보드만으로 담당 지역/제품의 ROI 상위·하위 세그먼트를 5분 이내 파악 가능해야 함.
- 월별 데이터 갱신 프로세스가 안정적으로 작동해야 함.

## 3. 범위 (Scope)

### 3.1 대상 사용자
- **주 사용자**: 영업 관리자 및 리더십 (지역/팀/제품 단위 성과 모니터링)
- **접근 권한**: 내부 영업팀 전체 접근 가능 (별도 역할 기반 접근제어 없음). 단, 의사 실명·NPI 등 식별정보가 포함되므로 사내 데이터 정책 및 Compliance 팀 검토를 별도 확인 권장.

### 3.2 핵심 기능 (In Scope)

| 기능 | 설명 | 우선순위 |
|---|---|---|
| 제품별 ROI 요약 | 제품별 총 지급액, 총 처방액(Tot_Drug_Cst), ROI 비율(처방액/지급액) 카드 및 랭킹 | P0 |
| Paid vs Unpaid 비교 | 지급 이력 유무에 따른 의사 그룹별 평균 처방 클레임 수·처방액 비교 (박스플롯/막대그래프) | P0 |
| Top HCP 리스트 | 처방액 기준, 지급액 기준 상위 의사 테이블 (검색·정렬 가능) | P0 |
| 제품명 드릴다운/필터 | 대시보드 전역에서 제품(Entresto, Cosentyx 등) 선택 시 모든 위젯이 해당 제품 기준으로 필터링 | P0 |
| 지급 유형별 분석 | Food&Beverage, Consulting Fee, Travel 등 지급 유형별 금액 분포 및 처방과의 상관성 | P1 |
| 전문과목별 분석 | 의사 전문과목(Specialty)별 처방/지급 패턴 비교 | P1 |
| 매칭/미매칭 의사 구분 표시 | 두 데이터셋에 모두 존재하는 1,677명과 한쪽에만 존재하는 의사를 구분하여 표시 | P1 |
| 지역(도시) 필터 | CA 내 도시 단위 드릴다운 | P2 |

### 3.3 제외 범위 (Out of Scope, v1)
- CA 외 타 주 데이터 확장 (현재 데이터 커버리지 부족)
- 컴플라이언스 이상탐지·알림 기능 (별도 요구 시 후속 버전에서 검토)
- 개별 MR(영업사원)의 방문 활동 로그 등 CRM 데이터 연동
- 예측 모델링 (처방 증가 예측 등)

### 3.4 데이터 정합성 이슈 (사전 확인 필요 사항)
- Open Payments의 24개 제품 중 Part D에 없는 19개 제품(Leqvio, Kisqali 등)은 ROI 계산이 불가하므로, 대시보드에는 "처방 데이터 매칭 가능 제품(5개)"과 "지급 데이터만 존재하는 제품"을 구분 표시해야 함.
- 지급 레코드는 제품이 최대 5개(product_1~5)까지 복수 매핑되므로, 지급액을 제품별로 배분할 때 단순 중복 집계가 되지 않도록 처리 방식 정의 필요 (예: 레코드당 제품 균등 분배 vs 전액 중복 반영).
- Part D의 `Tot_Drug_Cst`는 총 약제비이며 순수 매출/이익이 아님 — ROI 지표 정의 시 이 점을 명확히 주석 처리.

## 4. 핵심 지표 정의 (KPI Definitions)

| 지표 | 정의 | 산출식 |
|---|---|---|
| 제품별 지급액 | 특정 제품이 product_1~5 중 하나로 매핑된 지급 레코드의 payment_amount_usd 합계 | SUM(payment_amount_usd) WHERE product_n = 제품 |
| 제품별 처방액 | 해당 브랜드의 Tot_Drug_Cst 합계 | SUM(Tot_Drug_Cst) WHERE Brnd_Name = 제품 |
| ROI 비율 | 처방액 대비 지급액 배수 | 제품별 처방액 / 제품별 지급액 |
| Paid 의사 | Open Payments에 1건 이상 지급 기록이 있는 NPI | — |
| Unpaid 의사 | Part D에는 있으나 Open Payments에 매칭 기록이 없는 NPI | — |
| HCP당 평균 처방액 | Paid/Unpaid 그룹별 평균 | AVG(Tot_Drug_Cst) GROUP BY paid_flag |

## 5. 사용자 스토리

1. 영업 관리자로서, 담당 제품(예: Entresto)을 선택하면 해당 제품에 지급된 총액과 처방 매출, ROI를 한눈에 보고 싶다.
2. 영업 관리자로서, 지급을 받은 의사군과 받지 않은 의사군의 평균 처방량 차이를 비교해 지급 활동의 효과를 검증하고 싶다.
3. 영업 관리자로서, 처방액은 높지만 아직 지급 이력이 없는 의사 리스트를 확인해 영업 우선순위를 정하고 싶다.
4. 영업 관리자로서, 지급 유형(식사 vs 컨설팅 vs 강연)별로 어떤 유형이 처방과 더 강하게 연관되는지 확인하고 싶다.

## 6. 기술 요구사항

- **플랫폼**: 웹 기반 커스텀 대시보드 (특정 BI 툴 종속 없음)
- **데이터 소스**: 현재는 CSV 배치 업로드 방식. 향후 내부 데이터 웨어하우스 연동 여부는 별도 논의.
- **데이터 갱신 주기**: 월별 (Part D/Open Payments 실제 공개 주기와 별개로, 내부 처방 데이터 기준 월별 갱신 프로세스 수립 필요. Open Payments 공식 데이터는 연 1회 공개되므로 갱신 주기 불일치 가능성을 UI상에 "데이터 기준일" 표기로 보완)
- **성능**: 현재 데이터 규모(Part D 4천여 행, Open Payments 3.6만여 행) 기준 필터/드릴다운 응답 3초 이내
- **접근 제어**: 초기 버전은 별도 역할 기반 접근제어(RBAC) 없이 내부 영업팀 전체 오픈. 단, 로그인 인증(SSO 등 내부 표준)은 필요
- **데이터 프라이버시**: 의사 실명, NPI 등 식별정보 노출 범위에 대해 Compliance팀 사전 확인 권장 (Open Payments는 CMS 공개 데이터이므로 규제상 제약은 낮으나, 내부 처방 데이터 결합 시 사내 정책 재확인 필요)

## 8. 리스크

| 리스크 | 영향 | 대응 방안 |
|---|---|---|
| Paid vs Unpaid 비교가 인과관계가 아닌 상관관계임을 오해할 소지 | 영업 의사결정 왜곡 | 대시보드 내 "상관관계이며 인과관계 아님" 명시 문구 삽입 |
| Open Payments 연 1회 공개로 인한 월별 갱신 기대치 불일치 | 사용자 신뢰도 저하 | "데이터 기준일/최종 갱신일" UI 표기, 갱신 주기 사전 공지 |
| 19개 제품의 처방 데이터 부재로 ROI 계산 불가 | 분석 범위 제한 오해 | 제품 리스트에 "처방 데이터 없음" 배지 표시 |
| 식별정보(NPI, 실명) 오남용 가능성 | 컴플라이언스 리스크 | 접근 로그 기록, 사내 데이터 사용 정책 고지 |

## 9. 미결 사항 (Open Questions)

- Part D 데이터의 기준 연도가 Open Payments의 2022년과 동일한지 원 데이터 소스에서 확인 필요.
- 지급 레코드가 복수 제품에 매핑될 경우 금액 배분 방식(균등 분배 vs 중복 반영) 결정 필요.
- 향후 CA 외 지역 확장 계획 여부.
